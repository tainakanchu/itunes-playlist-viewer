# 選曲ワークスペースを初期化する (Crateforge)

**対話ヒアリング → 選曲方針の言語化 → ワークスペース雛形の生成** を行うスキル。
ここで作る `CLAUDE.md`（選曲ポリシー）は、`/dj-curator:build-set` が毎回読み込む土台になる。

## 役割 (重要)
- このスキルは **「DJ の嗜好・基準を引き出して言語化し、ワークスペースに定着させる」** ことに集中する。
- 選曲そのもの（候補プールの作成）はしない。それは `build-set` の仕事。
- 出力は **カレントディレクトリ** に置く（＝ユーザーが Claude Code を開いている選曲ワークスペース）。

## 前提
- **生成先はカレントディレクトリ**。意図した場所かを最初に確認する。
- Crateforge API は **任意**。繋がる場合はヒアリングを「実際に所有しているライブラリ」に基づけられる（推奨）。
  - ヘルパは同プラグイン同梱の `${CLAUDE_SKILL_DIR}/../build-set/scripts/crate-api.sh`（curl ラッパ）。
  - 繋がらなければ気にせずスキップし、手動ヒアリングだけで方針を作る。

## 手順

### 1. 初期化先の確認 / 既存検出
- cwd を初期化先として確認する（「ここを選曲ワークスペースにするね？」）。
- 既に `CLAUDE.md` や `sets/` がある場合は **上書きしない**。差分を提示し、追記/マージするかを確認する。
- `$ARGUMENTS` に方向性メモがあれば出発点として取り込む。

### 2. (任意) ライブラリを覗いてヒアリングを地に足つける
- API が繋がるなら `crate-api.sh GET /api/stats`（規模感）と `crate-api.sh GET /api/genres`（ジャンル token 頻度）を取得。
- 「あなたのライブラリ上位ジャンルはこれ。どれを軸に？どれは避ける？」と、**実データを見せて**選んでもらうと精度が上がる。

### 3. 対話でヒアリングする (一度に詰め込みすぎない)
ここで作るのは **持続的なプロファイル**（DJ の色・基準・運用則）。セットごとに変わる文脈（今日のスロット / 会場 / 尺 / ムード）は build-set が都度聞くので、ここでは **「よくやる現場の既定値」** として捉える。
**core を優先**して聞き、optional は後から育てる前提で空欄＋記入例を残してよい。ユーザーの言葉を尊重し、こちらの好みは押し付けない。

**① アイデンティティ / 音楽の色** (core)
- 名義・音楽的な色 / 攻め度の基調
- 主軸ジャンル・サブジャンルと、越境の流儀（地続きなら OK 等）
- シグネチャー曲（必ず候補に入れたい固定枠）〔opt〕

**② テンポ・キー・エナジーの流儀** (core)
- ホームテンポ＋標準 BPM レンジ＋漸進の仕方 / ハーフ・ダブルタイムを使うか〔opt〕
- キーへのこだわり度（例: 隣は Camelot 0・±1・A↔B が好み、+2 の energy boost は山場で）〔opt〕
- エナジー運用（±1 ずつ動かす / ピークは energy 高めを 1〜2 曲 / 谷・箸休めも作る）

**③ 声・言語・歌詞** (core)
- vocal / インスト比率の好み（例: 序盤は歌もの少なめ）
- **言語・地域の軸**（例「台語 / 華語 / 原住民語を区別」）＋ **捕捉レシピ**：言語などタグに無い属性は
  「代表アーティスト名 / その言語の特有語スイープ（例 台語＝毋・袂・媠・查某・暗暝…）/ ジャンル token 横断」で拾う ——
  **この語彙やコツ自体をプロファイルに書き残す**（次回以降の再利用資産になる）。
- 歌詞テーマ・露骨さの可否〔opt〕

**④ 母集団・品質・新鮮さ** (core)
- 既定レーティング下限（例 ★3＝`ratingMin=60`）＋ **★ の自己定義**（例: ★5＝実証済み / ★3＝試したい）〔opt〕
- 年代の好み・除外（音質が古すぎるもの等）〔opt〕
- 認知度バランス（クラウドプリーザー ⇄ ディグった深い曲。例 7:3）と、新鮮曲の仕入れ先〔opt〕
- 解析値の重視度（既定はメタデータ主体・BPM/Key/Energy はボーナス）

**⑤ よくやる現場の既定値**（per-set だが初期値として）(opt)
- よく立つスロット（オープン / ピーク / クロージング）
- 主な会場・イベント種別（クラブ / バー / フェス / 配信 / 結婚式…）
- 標準セット尺・曲数感 / 主な客層・時間帯

**⑥ 回避・重複・運用** (core)
- avoid（苦手ジャンル・使い古しの大曲・obvious 直球・露骨さ）
- 直近セット重複回避の窓（例: 過去 3 ヶ月の `sets/`）とローテーション可視化（`sets/` / play count）
- プレイリスト命名規則（例「<コンセプト> (叩き台)」）/ `sets/` 保存 / 候補プールは必要量の約 2 倍
- メタデータ外属性の `#tag` 運用（`comments` に `#台語` `#inst` 等 → `q=#台語` で検索可）〔opt〕

途中で **2〜4 行のサマリ**にまとめ、ユーザーに確認してから書き出す。

### 4. ワークスペース雛形を生成する
まず **方針の置き方（レイアウト）** を選んでもらう（`build-set` はどちらも読める）:
- **(A) シンプル**: 単一の `CLAUDE.md` に全方針。小さく始めたい人向け。
- **(B) モジュール (rules/)**: `rules/` に観点ごとの小さな Markdown を分割配置し、`CLAUDE.md` は薄いインデックス。
  方針が育つ・項目が多い人向け（Claude Code の `rules/` 慣習に沿う。1 ファイル 1 観点で見通しよく育てられる）。

共通の生成物（**既存は上書きしない**。あれば追記提案）:
- `README.md` — このワークスペースの使い方（build-set の起動例つき）。
- `sets/` — 過去の叩き台レポート置き場（空なら `.gitkeep`）。
- `inputs/`（任意）— フライヤー画像・テーマメモ置き場。

方針の中身（ヒアリング結果で埋める。**観点は共通**、置き方だけが A/B で違う）:

| 観点（手順3の ①〜⑥） | (A) `CLAUDE.md` の見出し | (B) `rules/` のファイル |
|---|---|---|
| ① アイデンティティ / 色・ジャンル・シグネチャー | 「DJ の色」 | `rules/00-identity.md` |
| ② テンポ・キー・エナジーの流儀 | 「テンポ・キー・エナジー」 | `rules/10-tempo-key-energy.md` |
| ③ 声・言語（＋捕捉レシピ）・歌詞 | 「声・言語」 | `rules/20-voice-language.md` |
| ④ 母集団・品質・新鮮さ（rating / 年代 / 認知度 / 解析） | 「母集団・品質」 | `rules/30-pool-quality.md` |
| ⑤ よくやる現場の既定値（スロット / 会場 / 尺 / 客層） | 「現場の既定値」 | `rules/40-context-defaults.md` |
| ⑥ 回避・重複回避・運用（命名 / `sets/` / `#tag`） | 「避けたい・運用」 | `rules/50-avoid-ops.md` |

- (A) の `CLAUDE.md` は `build-set/workspace-template/CLAUDE.md` と同じ体裁に揃える。
- (B) では各 `rules/*.md` を **1 観点・小さめ**に保ち、`CLAUDE.md` には「方針は `rules/` を参照」＋最重要サマリ数行だけ置く。
- どちらの場合も、ヒアリングで埋まらなかった観点は**プレースホルダ＋記入例**（コメント）を残し、後から育てられるようにする。

### 5. 仕上げ
- 生成物の一覧と `CLAUDE.md` の要点を提示する。
- 「方針はいつでも編集可。次は `/dj-curator:build-set <フライヤー / テーマ / 参照曲>` で叩き台を作れます」と案内する。

## 原則
- **破壊的操作はしない**（既存 `CLAUDE.md` / `sets/` を上書きしない。追記・マージは確認してから）。
- ユーザーの言葉・嗜好を最優先で言語化する。
- API が落ちていても **止まらない**（手動ヒアリングで方針だけ作る）。
- 方針は固定ではなく **育てる前提**。build-set を使ううちに気づいた嗜好を後から追記していい旨を伝える。
