# DJ 選曲の叩き台を作る (Crateforge)

**何らかのインプット → コンセプトのヒアリング → 候補プールの選定 → 新規プレイリスト化 → 根拠レポート** を行うスキル。

## 役割分担 (重要)
- AI は **「コンセプトの解釈」と「候補プールの選定」に集中**する。
- **曲順は人間が Crateforge の GUI で詰める**。だから**並べ替え (ordering) はしない**し、曲順最適化に深入りしない。出力は「いい感じの候補集合 = 叩き台」。
- 選曲は **メタデータ主体** (ジャンル / 年代 / レーティング / アーティスト等)。BPM / Key / Energy などの解析値は **「あれば使う」ボーナス**。未解析の曲も候補に入れてよい。

## 前提: Crateforge のローカル API
このスキルは Crateforge アプリの内蔵 HTTP API (既定 `http://127.0.0.1:8787`) を叩く。
ヘルパ: `${CLAUDE_SKILL_DIR}/scripts/crate-api.sh`（curl ラッパ。詳細なエンドポイントは `${CLAUDE_SKILL_DIR}/reference.md` を参照）。

**最初に必ず疎通確認する:**

!`"${CLAUDE_SKILL_DIR}/scripts/crate-api.sh" health`

- 失敗したら、ユーザーに「Crateforge アプリを起動し、設定 → 『AI 連携 / API』で API サーバーを有効化してください（既定ポート 8787）」と案内し、有効化を待ってから先に進む。
- 別ポートなら環境変数 `CRATEFORGE_API`（例 `http://127.0.0.1:9000`）を使う。
- WSL/Docker など**別ネットワークから叩く場合、`127.0.0.1` はホストと別物**。`health` が即失敗するときは、WSL2 をミラーモード化（`.wslconfig` に `networkingMode=mirrored` を書いて `wsl --shutdown`）するか、`CRATEFORGE_API=http://<ホストIP>:8787` を指定する。

## 選曲ワークスペース (嗜好・基準の置き場)
選曲の「方針・基準・嗜好」は **このスキルではなく、ユーザーが Claude Code を開いている作業ディレクトリ (選曲ワークスペース repo) の rules** に置く想定。
- まず **カレントディレクトリの `CLAUDE.md` / `rules/` / `README.md`** を読み、選曲方針（例: 既定の母集団は `ratingMin=60`(=★3 以上)、好みジャンル、避ける曲、DJ の色、過去セットとの重複回避など）を取り込む。
- 過去のセットが `sets/` にあれば参照し、重複を避ける材料にする。
- 選曲方針が見当たらない場合は、`/dj-curator:init-workspace` で対話初期化できることを伝える（`CLAUDE.md` か `rules/` を生成）。手動派には `${CLAUDE_SKILL_DIR}/workspace-template/` のテンプレートをコピーする方法も案内する。

## 手順

### 1. インプットを受け取り解釈する
`$ARGUMENTS` を確認する。**入力形式は問わない**:
- 画像パス（フライヤー）→ 画像を読み、イベント名・日時・出演・会場・雰囲気・ジャンル傾向を読み取る。
- テーマ / ムードのテキスト（例「夏の夕暮れの chill house」）→ そのまま意図として扱う。
- 参照曲 / 参照プレイリスト → 起点として似た方向を探る。
- 何も無ければ、まずインプットを尋ねる。

### 2. コンセプトをヒアリングする
**ワークスペースのプロファイル（`CLAUDE.md` / `rules/`）を土台に**、今回のセット固有の文脈を詰める（一度に詰め込みすぎない）:
- **スロット**（オープン / ピーク / クロージング）※最も選曲を左右する。warm-up ならピーク級を温存し energy 低めから。
- **会場・イベント種別**（クラブ / バー / フェス / 配信 / 結婚式…）と客層・時間帯。結婚式/企業は do-not-play 確認とリクエスト受容が既定。
- 尺・曲数感（例: 60〜90 分 ≒ 15〜25 曲）※候補は必要量の約 2 倍出してよい
- 全体のムード・物語（コンセプト文の核）/ 攻め度・テンポ帯・エネルギーの置き方
- 必ず入れたい曲・アーティスト / 今回の避けたい要素
- 母集団の絞り（レーティング下限・ジャンル・年代）※プロファイル既定を土台に上書き

ヒアリング結果を 2〜4 行の **コンセプト文**にまとめ、ユーザーに確認する。
（プロファイルが無ければ `/dj-curator:init-workspace` で先に作ると、以降このヒアリングが軽くなる）

### 3. ライブラリを把握して候補を集める
ヘルパ経由で API を叩く（`reference.md` 参照）。例:
- 全体像: `crate-api.sh GET /api/stats` / `crate-api.sh GET /api/genres`
- 母集団の収集（メタデータ主体）:
  - `crate-api.sh GET "/api/tracks?genre=House&ratingMin=60&yearFrom=2015&limit=300"`
  - フリーテキストや BPM/Key/Energy トークンは `q` に渡せる: `q=energy:0.6-0.9`、`q=bpm:120-126`
- 種曲から広げる（解析があれば）: `crate-api.sh GET "/api/tracks/<id>/similar?limit=20&keyCompatible=true"`
- **非 ASCII / 空白を含む検索**（曲名・アーティスト名・歌詞語など、日本語/中国語等）は `crate-api.sh GET /api/tracks q="<語>" genre=... limit=...` のように **`key=value` 形式**で渡す（`?...` に直書きすると URL エンコードされず壊れて全件返るので注意）。

複数の角度（ジャンル別・年代別・種曲の類似・レート上位）で広めに集め、重複を排除する。

#### メタデータに無い属性で絞りたいとき
言語（例: 台語 / 華語 / 原住民語）・ムード・年代の肌感など、**タグに無い属性**で母集団を作りたい場合は次の合わせ技が有効:
- **アーティスト軸** — その属性を代表するアーティストを名指しで `q=<アーティスト名>` 検索して引く。
- **特有語スイープ** — その言語/ムードに特徴的な語（例: 台語なら「毋/袂/媠/查某/暗暝」等）を `q` で順に当て、横断ヒットを集める。
- **ジャンル token 横断** — `GET /api/genres` で関連 token を洗い出し、`genre=<token>` で寄せて相互参照する。
- 判定がヒューリスティックになる場合は、根拠レポートに**確信度や判定根拠を正直に併記**する。

### 4. コンセプトに沿って候補を絞る
- コンセプト文・ワークスペース方針に照らして取捨選択する。
- **役割の素材が揃うように**広げる（オープナー級の落ち着き / コアのグルーヴ / 山場・アンセム / 箸休め / クローザー候補）。「全部バンガー」は避け、緩急の幅を持たせる（曲順は付けないが素材の多様性は選定品質）。
- **曲順は付けない**（人間が GUI で詰める）。ただし「外せない曲」や「フックになりそうな曲」は根拠で触れてよい。
- 解析値があれば補助的に使う（BPM 帯のまとまり、Key 互換、Energy のレンジ）。無くても無理に解析を要求しない。

### 5. 新規プレイリストを作る
- 名前はイベント名やコンセプトから（例「夏の夕暮れ chill house (叩き台)」）。
- 作成: `crate-api.sh POST /api/playlists '{"name":"...","isFolder":false}'` → 返り値の `playlistId` を取得。
- 曲を追加: `crate-api.sh POST /api/playlists/<playlistId>/tracks '{"trackIds":[...]}'`（`trackId` は手順 3〜4 で得た曲の `trackId`）。
- 反映確認: `crate-api.sh GET /api/playlists/<playlistId>/tracks`。

### 6. 根拠レポートを返す
**Markdown** で以下を返す（チャットに表示し、ワークスペースに `sets/` があれば「保存しますか？」と提案して `sets/<日付>-<イベント>.md` に書く）:
- セットのコンセプト（2〜4 行）
- 母集団の絞り条件（rating / genre / 年代など、実際に使ったフィルタ）
- 選曲リスト：各曲に 1 行の根拠（なぜこのコンセプトに合うか。分かる範囲で BPM/Key/Energy/ジャンル/年代に触れる）
- 補足：外した有力曲や、人間が GUI で曲順を詰める際のヒント（フック候補・エネルギーの山場になりそうな曲など）

最後に「**プレイリストは Crateforge アプリに作成済みです。曲順はアプリ上で調整してください**」と伝える。

## 原則
- 破壊的操作はしない（既存プレイリストの削除・改変はしない。常に**新規**プレイリストを作る）。
- API が落ちている / 未解析が多い等は、止まらず可能な範囲で進め、状況を正直に伝える。
- ユーザーのワークスペース方針を最優先する（このスキルの既定より rules を優先）。
